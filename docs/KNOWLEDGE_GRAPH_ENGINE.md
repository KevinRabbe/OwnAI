# OwnAI Knowledge Graph Engine

The Knowledge Graph Engine connects OwnAI's memory, tasks, patches, files, skills, replay lessons, confidence signals, and engineering decisions into a relationship-aware system.

## Core Principle

```text
OwnAI should understand not only things, but how things are connected.
```

A file, task, bug, patch, replay lesson, and skill should not live as isolated records. Their relationships should become queryable and visible.

---

# Main Goals

- Connect files, systems, tasks, patches, bugs, lessons, skills, and decisions
- Support temporal memory navigation
- Improve impact analysis
- Improve thread planning
- Improve confidence scoring
- Improve replay and patch quality analysis
- Support self-improvement safety
- Enable relationship-based Telescope navigation

---

# Graph Entities

## File

Represents source files, docs, config files, tests, and generated artifacts.

Example:

```json
{
  "type": "file",
  "id": "file:packages/memory-system/src/memory.ts",
  "path": "packages/memory-system/src/memory.ts",
  "language": "typescript"
}
```

---

## System

Represents OwnAI systems or project subsystems.

Examples:

```text
memory-system
heatmap-engine
telescope-engine
microscope-engine
context-replay-engine
```

---

## Task

Represents a user task or queued task.

---

## Patch

Represents a patch record generated by the Git Engine or future patch systems.

---

## Skill

Represents reusable skill workflows.

Examples:

```text
analyze_repo
run_verification
fix_bug_with_tests
compress_context
```

---

## Lesson

Represents replay lessons, failed attempts, successful strategies, or project rules.

---

## Decision

Represents engineering decisions, such as:

```text
Use JSON for warm memory.
Use SQLite for long-term memory.
Prefer minimal sufficient patches.
Cloud is optional, not required.
```

---

# Relationship Types

```text
file BELONGS_TO system
task TOUCHED file
patch MODIFIED file
patch CAUSED verification_result
skill USED_DURING task
lesson DERIVED_FROM replay_report
decision AFFECTS system
system DEPENDS_ON system
bug OCCURRED_IN file
replay OPTIMIZED task
confidence DROPPED_BECAUSE_OF file
```

# Storage Strategy

Initial implementation can use SQLite.

Suggested storage:

```text
.ownai/memory/long_term.db
```

Core tables:

```text
nodes
edges
timeline_events
```

Example schema concept:

```sql
CREATE TABLE nodes (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  label TEXT NOT NULL,
  metadata_json TEXT,
  created_at TEXT NOT NULL
);

CREATE TABLE edges (
  id TEXT PRIMARY KEY,
  source_id TEXT NOT NULL,
  target_id TEXT NOT NULL,
  type TEXT NOT NULL,
  weight REAL DEFAULT 1.0,
  metadata_json TEXT,
  created_at TEXT NOT NULL
);
```

# Telescope Integration

The Telescope Engine can use the graph to navigate:

```text
project
→ system
→ task cluster
→ patch history
→ exact files
```

This enables questions like:

```text
What did we work on 4 months ago?
Which patches affected the Memory System?
What tasks are connected to replay optimization?
What bugs keep returning in this area?
```

---

# Microscope Integration

The Microscope Engine can zoom from a graph node into:

```text
- exact files
- exact functions
- exact patches
- exact reports
- exact replay lessons
```

---

# Confidence Integration

The Confidence Engine can use graph signals:

```text
- file historically caused failures
- system has many related failed attempts
- patch touches high-risk graph cluster
- task is similar to previous successful tasks
```

---

# Thread Coordinator Integration

The Thread Coordinator can use graph overlap to avoid conflicts.

Example:

```text
Task A touches memory-system.
Task B also touches memory-system.
Do not run in parallel unless worktrees and merge risk are acceptable.
```

---

# Dashboard Integration

The UI can show:

- hot system clusters
